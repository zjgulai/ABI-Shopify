# ABI 网站体验审计 2026-06-28

## 1. 审计范围

事实：
- 本轮审计对象为当前项目本地静态站点：`/Users/pray/project/shopify/kb/site`。
- 本地预览地址为 `http://127.0.0.1:8765`。
- 页面数据来自 `kb/site/kb_data.js`，当前统计为：`14` 个流程节点、`558` 个 RAG chunks、`253` 个 KG entities、`776` 个 KG relations。
- 已截图覆盖首屏架构、流程节点、经营作战台、工程仓库地图、增长作战室、Agent 工作流、风险闸口、信息源覆盖、工具包入口、问答区和移动端视口。

推断：
- 当前网站已经从“知识库索引页”升级为“ABI 智能独立站作战台”的雏形，但核心交互仍偏展示型，尚未完全变成可选择路径、可下钻 SOP、可判断风险状态的操作台。

不确定项：
- 本轮未访问线上部署页面，也未调用外部 provider。
- 本轮未对真实 Shopify 测试店执行写入或授权检查。

## 2. 当前价值判断

### Step 1: 首屏与产品定位

健康度：中高。

事实：
- 首屏清楚表达 `ABI 智能化独立站 · Shopify AI 经营知识库`。
- 首屏有 RAG/KG/节点数量等量化资产信号。
- 当前视觉语气偏“知识系统 + 技术资产展示”，具备可信度。

风险：
- 对首次进入的业务用户来说，“下一步该怎么走”不够明确。
- 首屏没有按照经营目标给出路径选择，例如新店冷启动、Headless 改造、站外增长放量、T7 测试店自动化。

建议：
- 将首屏升级为 `Command Center`，优先展示“选择你的目标 -> 推荐路径 -> 关键 SOP -> 风险闸口”。

### Step 2: 流程节点专题

健康度：中。

事实：
- 当前有 `00-10` 与 `90-92` 横切层节点。
- `04-内容素材生产`、`08-订单履约供应链`、`09-客户运营会员` 当前文档数为 `0`。
- 节点卡片已可展开，能够展示说明、文档数、chunk 数、知识图谱实体数和关系数。

风险：
- 节点卡片本身是知识目录，但还没有转成“节点任务包”：输入、输出、SOP、依赖、验收、风险。
- 文档数为 `0` 的节点如果不显式标为内容债，容易让用户误以为已覆盖。

建议：
- 为每个节点增加 `node_readiness`：`ready / partial / content_debt / blocked_by_external_auth`。
- 将节点详情扩展为 `输入 -> 操作 -> 产出 -> 证据 -> 人审闸口`。

### Step 3: 经营作战台

健康度：中高。

事实：
- 经营作战台已经覆盖诊断、建站、内容、增长、转化、自动化等主题。
- 模块语言比普通知识库更接近经营动作。

风险：
- 卡片信息丰富，但没有主按钮或决策分支。
- 用户无法快速回答“我现在处于哪个阶段，该先做什么”。

建议：
- 增加 `Use-case Journey Selector`：
  - 新店 0-1 冷启动
  - 老店 CRO 提升
  - Headless / Hydrogen 技术改造
  - 站外全域增长
  - AI Agent 自动化编排
  - T7 测试店受控写入准备

### Step 4: 工程仓库地图与技术选型

健康度：中高。

事实：
- 当前工程仓库地图展示 Theme、Hydrogen、App Template、UI Extension、CLI / MCP 五类高相关项目。
- 已有适用场景、核心价值、可复用能力和风险说明。

风险：
- 所有技术路线的视觉权重接近，缺少“按业务问题选型”的引导。
- 非技术用户可能无法判断 Theme、Hydrogen、App、MCP 的边界。

建议：
- 新增 `Technical Decision Center`：
  - 当前店铺阶段
  - 是否需要 Checkout / Admin / Storefront API
  - 是否需要 Headless 性能和自定义体验
  - 是否涉及生产写入
  - 推荐技术路径与风险闸口

### Step 5: 增长作战室

健康度：中。

事实：
- 当前增长作战室覆盖 SEO/GEO、Meta/Google Ads、Reddit GEO、YouTube/PR、KOL/UGC、Affiliate/Creator Store Front、Deal/大促。

风险：
- 渠道模块还没有统一成“流量意图 -> 承接页面 -> 素材 -> 归因 -> 复盘”的闭环。
- 平台算法、社区规则、GEO/SEO 等时效性内容需要持续标记 `needs_external_verification`。

建议：
- 按渠道增加 SOP 模板：
  - 渠道角色
  - 适合产品阶段
  - 内容资产要求
  - 落地页承接
  - 归因指标
  - 风险闸口

### Step 6: Agent 工作流库

健康度：中高。

事实：
- 当前已展示市场洞察、内容素材、红人/联盟、SEO/GEO、广告诊断、经营复盘六类 Agent。
- 与 `10-自动化编排` 蓝图一致，具备后续自动化扩展基础。

风险：
- Agent 仍偏能力清单，还没有呈现为可执行任务单。
- 每个 Agent 的输入、输出、人审点、禁区和调用权限没有在页面层形成统一结构。

建议：
- 为 Agent 增加 `Agent Card 2.0`：
  - 任务目标
  - 输入数据
  - 可调用工具
  - 产出物
  - 人审节点
  - 禁止动作
  - 适配节点

### Step 7: 风控与人审闸

健康度：高。

事实：
- 当前已覆盖真实店铺写入、Admin API/MCP token、Checkout/UI Extension、社区发帖、UGC/KOL 素材复用、广告预算/折扣佣金。
- 与 T7 测试店 Runbook 的边界一致。

风险：
- 页面未将“当前不能做什么”和“满足什么条件后才能做”放到足够突出的执行状态层。

建议：
- 新增 `Execution Readiness Center`：
  - 本地可做
  - 需要用户授权
  - 需要测试店
  - 禁止生产写入
  - 需要外部平台核验

### Step 8: 信息源可信度与覆盖矩阵

健康度：中高。

事实：
- 当前页面展示 10 类信息源。
- 覆盖矩阵能体现各节点资料覆盖度。

风险：
- 宽表对移动端和快速阅读不够友好。
- `needs_external_verification`、`local_material_only`、`official_verified` 等状态需要被用户一眼识别。

建议：
- 改成可筛选的 `Evidence & Trust Center`：
  - 按节点筛选
  - 按来源类型筛选
  - 按验证状态筛选
  - 展示证据链和更新时间

### Step 9: 搜索与问答

健康度：中。

事实：
- 当前站点提供搜索和 DeepSeek API Key 手动录入问答。
- 已符合“不在环境变量硬编码 provider key”的方向。

风险：
- 搜索和问答与具体模块之间联系不够强。
- 用户在某张卡片上产生的问题，不能直接带上下文进入问答。

建议：
- 为每个模块增加 `Ask about this` 查询入口。
- 增加预设查询：
  - 我是新店冷启动，下一步做什么？
  - Theme 与 Hydrogen 如何选择？
  - T7 为什么还不能执行真实写入？
  - 哪些内容需要外部核验？

### Step 10: 移动端与可访问性

健康度：中。

事实：
- 移动端截图显示主体内容可读，顶部保留品牌与 DeepSeek 按钮。
- 桌面截图中的主要模块均能渲染。

风险：
- 移动端导航链接隐藏后，没有看到替代菜单入口。
- 复杂表格、长卡片和密集标签在移动端可能需要折叠策略。
- 键盘焦点样式和卡片可点击语义需要进一步验收。

建议：
- 增加移动端菜单按钮。
- 为表格提供卡片化移动布局。
- 对所有按钮、输入框、卡片详情入口补充清晰 focus 样式。

## 3. 结论

事实：
- 当前站点已经具备“知识资产展示 + 经营模块 + 工程模块 + 风控模块”的基础。
- 数据层已经有足够资产支持 PRD 2.0 扩展。

推断：
- 下一阶段最有价值的增量不是继续堆更多卡片，而是把现有内容组织成“目标路径、技术选型、SOP 执行、证据可信、风险状态”的操作闭环。

不确定项：
- 是否优先服务业务经营者、技术执行者，还是二者同权，需要产品决策确认。
- 是否把 T7 测试店准备状态做成显性页面模块，需要确认，因为目前缺少 Shopify 测试店授权。
