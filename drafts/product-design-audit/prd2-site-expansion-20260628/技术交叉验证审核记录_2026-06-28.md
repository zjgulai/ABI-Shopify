# PRD 2.0 技术交叉验证审核记录

## 1. 本地代码事实

验证对象:
- `kb/site/index.html`
- `kb/site/kb_data.js`
- `kb/_build/build_site_data.py`

结论:
- 当前站点是静态优先单页站点,页面模块由 `kb_data.js` 数据驱动。
- `build_site_data.py` 当前已生成 `sourceRegistry`、`coverageMatrix`、`repoPlaybooks`、`growthPlaybooks`、`agentWorkflows`、`riskGates`、`skillCatalog`。
- 当前数据统计为 `14 nodes / 558 chunks / 253 entities / 776 relations`。
- 当前 0 文档节点为 `04-内容与素材生产`、`08-订单履约与供应链`、`09-客户与会员运营`。

对 PRD 的影响:
- PRD 2.0 采用单页锚点式 MVP,不要求立即迁移前端框架。
- 新增页面区块均以 `build_site_data.py -> kb_data.js -> index.html` 为落地路径。
- 内容债必须成为页面显性模块。

## 2. KB 与 Runbook 事实

验证对象:
- `kb/PRD_ABI智能化独立站.md`
- `kb/10-自动化编排/全自动运营蓝图.md`
- `kb/10-自动化编排/AI-Toolkit_UCP测试店受控写验收Runbook.md`
- `kb/迭代方案与PRD_TODO.md`

结论:
- ABI 已有 00-10 与 90-92 节点产品骨架。
- 自动化蓝图明确三层架构:编排层、能力层、数据层。
- T7 当前状态是测试店授权前置,真实读写待用户授权与人审。
- DeepSeek API Key 当前采用页面手动录入方式,站点 README 已明确不默认写服务器环境。

对 PRD 的影响:
- PRD 2.0 不把 T7 设计成执行台,只设计 readiness 和 Runbook 表面。
- 所有真实店铺写操作都必须显示测试店、人审、回滚和证据要求。

## 3. Shopify 官方技术面

核验范围:
- Hydrogen / Storefront API:适合 Headless Storefront,不能替代 Admin 写操作。
- Admin GraphQL API:面向店铺管理与 mutation,必须有 token、scope 和目标店铺边界。
- Shopify Functions / Checkout UI extensions:适合结账、折扣、校验等 surface,需要 target、版本和测试店验证。
- Shopify CLI / App 模板 / AI Toolkit / Dev MCP:适合开发、脚手架和 Agent 接口,但不等于可绕过权限。
- Storefront MCP / UCP readiness:适合后续 AI channel 与买家侧能力准备,当前仍需按最新官方入口复核。

官方参考:
- https://shopify.dev/docs/storefronts/headless/hydrogen
- https://shopify.dev/docs/api/admin-graphql
- https://shopify.dev/docs/api/storefront
- https://shopify.dev/docs/api/functions
- https://shopify.dev/docs/apps/build/checkout
- https://shopify.dev/docs/apps/tools/cli
- https://shopify.dev/docs/apps/build/ai-toolkit
- https://shopify.dev/docs/apps/build/storefront-mcp

对 PRD 的影响:
- Technical Decision Center 必须区分 Theme、Hydrogen、App、Extension、CLI/MCP、Storefront MCP/UCP。
- Execution Readiness 必须区分 Storefront 买家侧能力与 Admin 店铺管理能力。
- Checkout、Functions、Admin API 相关能力必须默认进入人审或测试店状态。

## 4. Product Design 审计事实

验证对象:
- `screenshots/01-hero-architecture.png` 至 `screenshots/12-mobile-workbench.png`
- `ABI网站体验审计_2026-06-28.md`

结论:
- 当前页面已具备知识资产展示、工程模块、增长模块、Agent 模块和风控模块。
- 首屏仍偏介绍型,用户路径选择不够突出。
- 移动端需要菜单入口与表格移动降级。
- Search/Ask 与模块上下文连接不足。

对 PRD 的影响:
- 首屏改为 Command Center。
- 新增 Journey Selector。
- 每个 Journey、SOP、Agent、风险闸口都需要 `Ask about this`。
- 移动端导航与 focus 样式列为 P0。

## 5. 审核结论

PRD 2.0 可落地条件成立:
- 现有数据生成管线可承载新增数据对象。
- 现有静态页面结构可承载新增区块。
- 当前知识库已经有足够内容支撑 P0 页面改造。
- T7 等未来能力有明确边界,不会被设计成当前可执行动作。

剩余前置:
- 执行开发前仍需确认 P0/P1/P2 优先级。
- 若要进入 T7 真实验证,必须另行完成测试店授权与逐次人审。
